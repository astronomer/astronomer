## Goal

For every `Airflow` CR running in the customer's cluster, create a corresponding `Deployment` record in Houston such that:

1. The CR's name, namespace, executor, runtime version, replicas, env vars, and DAG-deployment method are reflected in Houston.
2. The deployment is attached to a workspace, with the right RBAC bindings.
3. The customer's existing Airflow users (if any) are mapped to APC users with appropriate roles.
4. The customer can log into Astro UI and immediately see and operate the deployment.
5. Anything that doesn't cleanly map to Houston's data model is preserved (in a JSON column) rather than dropped.

## Proposed approach

### Phase A — Catalogue an existing `Airflow` CR

Given a CR (read from the cluster via Commander's new `ListCustomResources` RPC from [Task 2](m2-task-2-connect-operator-to-commander.md)), reduce its fields into Houston's deployment-creation input. A first cut of the mapping table:

| CR field | Houston field | Notes |
|----------|---------------|-------|
| `metadata.name` | `releaseName`, `label` (default) | |
| `metadata.namespace` | `namespace` | |
| `spec.executor` | `executor` arg to upsert | One of CeleryExecutor, KubernetesExecutor, etc. |
| `spec.image` | `image` arg | May include a customer registry — preserve. |
| `spec.runtimeVersion` | `runtimeVersion` | |
| `spec.airflowVersion` / runtime → AF version mapping | `runtimeAirflowVersion` | _Mapping logic TBD — currently in [`airflow-operator/pkg/versions/`](../../../../airflow-operator/pkg/versions/)._ |
| `spec.scheduler` | `scheduler` arg | replicas, resources, env. |
| `spec.workers[]` | `workers` arg | List → first/default worker group; extra groups → `config` JSON. |
| `spec.triggerer` | `triggerer` arg | replicas, resources. |
| `spec.webserver` / `spec.apiServer` | `webserver` / `apiServer` args | Pick based on AF 2.x vs 3.x. |
| `spec.dagDeployment` | `dagDeployment` arg | Map operator's DAG method to APC's. _Open question — see below._ |
| `spec.pgbouncer` | `pgbouncerConfig` | |
| `spec.redis` | _no first-class Houston field_ | Store in `config` JSON. |
| `spec.statsd` | _no first-class Houston field_ | Store in `config` JSON; replicate as chart default if absent. |
| `spec.metadataConnection` / `spec.resultBackendConnection` | `metadataConnection`, `resultBackendConnection` | Match data types. |
| Customer labels / annotations on the CR | `config.customerMetadata` (new JSON sub-key) | |
| Customer env vars (`spec.airflow.config.env`) | `environmentVariables` arg | |

Anything not in this table goes into `Deployment.config.adoption.rawCRSnapshot` (JSON), and we file a follow-up to add a first-class column.

### Phase B — Discover and propose

A new GraphQL query and CLI command perform discovery without committing:

- New query: `discoverOperatorDeployments(clusterId: ID!, namespacePattern: String, labelSelector: String): [DiscoveredDeployment!]!`
  - Houston calls Commander's `ListCustomResources` and runs the catalogue mapping.
  - Returns each candidate plus a compatibility report: `compatible | partial | incompatible`, and per-field flags.
- New CLI: `astro deployment discover --cluster <id> --namespace <pattern>` — prints the same output.
- The CRD spec → Houston field mapping logic lives in **one** place. _Open question:_ Houston-side (TypeScript) or Commander-side (Go)? Same call-site question as M3 / Task A.

### Phase C — Adopt

New GraphQL mutation: `adoptDeployment(input: AdoptDeploymentInput!): Deployment!` — or extend `upsertDeployment` with an `adoptFromCR: ID` argument (per the milestone description: *"upsert deployment with some flag called migrate"*).

Either way the resolver:

1. Validates the CR exists in the cluster (calls Commander).
2. Resolves / creates a workspace per `workspaceId` (or default workspace if not provided).
3. Builds the upsert payload using Phase A's mapping.
4. Marks the deployment as adopted (likely a flag inside the `config` JSONB column — see Task 2 Option B).
5. Inserts the `Deployment` row **without** firing the standard create-worker that would `ApplyCustomResource` over the existing CR. Either:
   - Skip the publish, or
   - Publish a new event `DEPLOYMENT_ADOPTED_FROM_CR` that the workers know to handle differently.
6. Returns the new Deployment record.

The skipped-publish path is safer initially; the dedicated event lets us evolve later.

### Phase D — User and role migration

The operator doesn't own user management — Airflow's own RBAC (webserver users) is what exists today. The PRD positions APC's RBAC layer as additive. Two flows:

1. **System admin bootstrap.** As part of CP install (M2 Task 1 if unified, or already-existing if split), a system admin login is provisioned. _(Houston has a bootstrap path — verify mechanism: env-var / CLI / first-login-wins.)_ This admin runs the adoption flow.
2. **Airflow → APC user migration.** Manual or scripted. For each Airflow user, create an APC user (`inviteUser` mutation from `mutation/index.js:31`), attach to a workspace, give a workspace/deployment role from the enum.

Sub-questions (from the milestone description's bullets):

- **Mapping Airflow roles → APC roles.** Airflow's defaults (`Admin`, `Op`, `User`, `Viewer`, `Public`) don't 1:1 with `WORKSPACE_ADMIN` / `DEPLOYMENT_*`. A reasonable default mapping is _TBD_ — needs product input.
- **Bulk import.** `inviteUser` is per-user. For a customer with hundreds of users, a CSV / script wrapper is desirable. Possibly a new mutation `bulkInviteUsers`, or a CLI utility.

### Phase E — Verification

- Adopted deployment shows in `listDeployments` with adopted indicator.
- Adopted deployment shows in the UI with a banner explaining limitations (per PRD CUJ 4).
- Editing replicas through the UI propagates through Task 2's connected-Commander path.
- DAGs continue to run.

## Open questions

- [ ] **Workspace assignment.** Auto-assign to a "default" workspace, or require the customer to pick during adoption? Multiple adopted deployments — same workspace or per-team?
- [ ] **System admin bootstrap.** How is the first system admin created during DP-only install? In unified mode the existing path applies; split mode is more nuanced.
- [ ] **Airflow user import scope.** Airflow's webserver users live in the Airflow metadata DB, not the CR — does APC fetch them, or is user migration entirely manual / one-off?
- [ ] **Airflow roles → APC roles mapping.** Default mapping needs product input.
- [ ] **CR fields without a Houston column.** Many `Airflow.spec` fields (e.g. custom env, custom pod template, multiple worker groups, statsd config) don't have first-class Houston columns. Are we fine living in `config` JSONB, or do we want a structured migration to add columns? Affects DB migrations.
- [ ] **DAG deployment method.** Operator supports image-baked DAGs, DAG-only deploys (via APC's DAG server), git-sync. Mapping CR DAG-deployment configuration into Houston's `dagDeployment` arg requires fidelity — see also [`../03-gap-analysis.md`](../03-gap-analysis.md) Gap 10 / Gap 11 / Gap 12.
- [ ] **What happens if the CR is later edited by `kubectl`?** Does Houston re-sync, drift-detect, or warn?
- [ ] **Reversibility.** Can a customer "un-adopt" a deployment back to operator-only control? Useful for trial-period semantics.
- [ ] **Image / registry preservation.** Customer's CR may reference their own registry. After adoption, do image upgrades flow through the APC registry mirror, or stay on the customer's source?
- [ ] **License / billing counter.** From the PRD: does an adopted deployment increment the APC license counter? Need answer before GA.
