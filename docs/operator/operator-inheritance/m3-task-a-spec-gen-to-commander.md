# M3 / Task A — Move operator CRD spec generation from Houston to Commander

**Parent milestone:** [M3 — Stage 2 parity](m3-stage-2-parity.md)
**Linear issue:** _not yet filed_
**Owner:** _TBD_
**Effort:** _TBD (likely L)_

> Linked Linear milestone description (verbatim):
> *"Changes in APC 1.0+: The responsibility of spec generation and applying it lied on the houston (CP service). After 1.0, the responsibility should be shifted to commander (DP service) since the Operator CRD will be applied in the DP. So we need to migrate operator related code to commander."*

---

## Goal

Today: Houston (CP) builds a complete Airflow CRD spec from helm values + deployment record, JSON-serializes it, ships it across to Commander (DP) via gRPC, and Commander does an unstructured apply. Operator CRDs are DP-side resources, so the spec generation belongs there.

After this task: Houston only ships the deployment's "intent" (logical fields) to Commander; Commander reads helm values from its own cluster, fills defaults, and constructs the CRD spec locally before applying.

### Current gRPC contract

Houston→Commander payload, per [`commander/_proto/custom_resource.proto:9-25`](../../../../commander/_proto/custom_resource.proto#L9-L25):

```
ApplyCustomResourceRequest {
  string spec = 1;            // full CR JSON, generated by Houston
  string namespace = 2;
  string apiVersion = 3;      // "airflow.apache.org/v1beta1"
  string kind = 4;            // "Airflow"
  string name = 5;            // releaseName
  string correlation_id = 7;
  repeated Label namespaceLabels = 8;
  repeated Annotation namespaceAnnotations = 9;
  string resourceGroup = 10;
  string resourceVersion = 11;
  string resourcePlural = 12;
  repeated Secret secrets = 13;
  repeated ConfigMap configMaps = 14;
  string k8sManifests = 15;   // additional K8s YAML, e.g. DAG server
  string dagDeploymentType = 16;
}
```

Commander side implementation (per recon):

- Handler: [`commander/api/custom_resource.go:10`](../../../../commander/api/custom_resource.go#L10) → `provisioner.ApplyCustomResource(in)`.
- Kubernetes apply: [`commander/kubernetes/custom_resource.go:18-49`](../../../../commander/kubernetes/custom_resource.go#L18-L49) — fetch-then-create-or-update; spec is treated as opaque JSON.
- The provisioner package (`commander/provisioner/kubernetes/kubernetes.go:1333` — per the M3 recon) deserializes the JSON, creates secrets/configmaps, then calls `k.kube.Resource.ApplyCustomResource()` to apply.

Commander has no current access to: a Helm chart loader (it does have one for helm-mode deploys — [`commander/helm/client.go`](../../../../commander/helm/client.go), ~595 lines), runtime-version metadata (no `updates.astronomer.io` calls), or Houston DB. It's stateless for CRD purposes.

## Proposed end state

After this task:

1. Spec generation lives in Commander (Go). The new function takes the deployment's logical config (executor, replicas, env, runtime version, DAG method, …) and returns the unstructured Airflow CR.
2. Houston only emits the logical config. The gRPC payload becomes much smaller; the giant JSON `spec` field goes away or becomes optional for transition.
3. Houston's `houston-api/src/lib/deployments/operator/` directory shrinks: the orchestration and worker plumbing stays; the spec-construction internals leave.
4. Commander gains read-access to whatever it needs to materialize defaults — likely chart values mounted at install time, plus runtime-version metadata.

## Proposed approach

### Step 1 — Define the logical contract

A new proto message — call it `OperatorDeploymentIntent` — captures everything Houston needs to send. Initial sketch:

```proto
message OperatorDeploymentIntent {
  string release_name = 1;
  string namespace = 2;
  string executor = 3;               // CeleryExecutor | KubernetesExecutor | LocalExecutor | AstroExecutor
  string runtime_version = 4;
  string airflow_version = 5;        // derived if absent
  string image = 6;
  Scheduler scheduler = 7;
  repeated WorkerGroup workers = 8;
  Triggerer triggerer = 9;
  Webserver webserver = 10;          // for AF 2.x
  ApiServer api_server = 11;         // for AF 3.x
  DagDeployment dag_deployment = 12;
  map<string, string> env = 13;
  // … and so on
}
```

The shape mirrors the parameters Houston already collects in [`upsertDeployment`](../../../../houston-api/src/schema/mutation.js#L683-L792). One-to-one is intentional — this is a translation, not a redesign.

### Step 2 — Port the spec generator to Go

In Commander, mirror the structure of `houston-api/src/lib/deployments/operator/index.js`:

| Houston (JS) | Commander (Go) — proposed |
|---|---|
| `getCRDSpecFromHelmValues()` | `internal/operator/spec.go: BuildAirflowSpec(intent)` |
| `getSecretsForOperatorMode()` | `internal/operator/secrets.go: BuildSecrets(intent)` |
| `getConfigMapsForOperatorMode()` | `internal/operator/configmaps.go: BuildConfigMaps(intent)` |
| `addSchedulerComponentToSpec()` etc. | `internal/operator/components/*.go` |
| `manifests/dag-only-deploy/*.hbs` | `internal/operator/manifests/dag-only-deploy/*.yaml.tmpl` (Go templates) |

Tests in `houston-api/src/lib/deployments/operator/index.unit.test.js` (~6359 lines) are the **single most valuable artefact** for this task — they define the existing behavior. Port them as table-driven Go tests with the same input/output fixtures.

### Step 3 — Wire chart values into Commander

Commander needs the operator-defaults that today live in the houston configmap (per [`astronomer/charts/astronomer/templates/houston/houston-configmap.yaml:99-101`](../../../charts/astronomer/templates/houston/houston-configmap.yaml#L99-L101)). Two approaches:

| Approach | Pros | Cons |
|---|---|---|
| (1) Mount a commander-configmap with operator defaults at install time | Familiar, one-time chart change | Doubles the "where do these defaults live" question; risk of drift with houston-configmap |
| (2) Have Commander pull defaults from Houston via gRPC at startup or on-demand | Single source of truth | Adds a CP→DP dependency, splits across processes |

(1) is simpler and matches how Commander already gets its config. The houston-configmap operator stanza moves to (or is duplicated in) a commander configmap. [`astronomer/charts/astronomer/values.yaml`](../../../charts/astronomer/values.yaml) gets a new `commander.operator.defaults` block, and we render it into a new commander-mounted configmap. Houston-side reads can deprecate after the migration is complete.

> Use the `env-readme-updater` skill when changing `commander/config/config.go` for new env vars and the `grpc-readme-updater` skill for proto changes.

### Step 4 — Switch the call site behind a feature flag

Phased rollout:

1. Add `OperatorDeploymentIntent` proto + new RPC `ApplyOperatorDeployment` while keeping `ApplyCustomResource` intact.
2. Implement Commander-side spec gen.
3. Add a Houston-side feature flag (env var or config) `OPERATOR_SPEC_GEN_LOCATION` ∈ `{houston, commander}`. Default `houston`.
4. Switch the workers to call the new RPC when the flag is `commander`. The old RPC path is the fallback.
5. In CI/local-dev, flip the flag and run the full integration suite.
6. After a release of soak time, flip the default. Remove the Houston-side code in a subsequent release.

### Step 5 — Migration of in-flight deployments

For APC clusters upgrading from a release that used Houston-side spec-gen to one that uses Commander-side spec-gen: existing CRs are already applied and have no `houston.io/spec-source` annotation. We can either:

- **Big-bang re-emit** on first reconcile: Houston regenerates the intent, Commander regenerates the spec, Commander does an apply. Risk: any drift gets normalized, possibly stomping customer edits (see [M2 Task 2](m2-task-2-connect-operator-to-commander.md) — same hazard).
- **Lazy** — only emit when Houston is asked to update the deployment.

The PRD's "observe and augment" philosophy argues for lazy; this stays consistent with adoption semantics.

## Open questions

- [ ] **Source-of-truth for defaults.** One configmap, two configmaps mirrored, or commander pulls from Houston at startup?
- [ ] **Runtime version metadata.** Today Houston resolves runtime → AF version mappings using its own runtime list. Commander would need similar — fetch from `updates.astronomer.io`? Mount a static manifest? Reuse the operator's own version pkg ([`airflow-operator/pkg/versions/`](../../../../airflow-operator/pkg/versions/))?
- [ ] **DAG server templates.** Where do the `manifests/dag-only-deploy/` templates ultimately live — commander, or the airflow-operator subchart? Aligning these with Gap 10 ([`../03-gap-analysis.md`](../03-gap-analysis.md)) is worth considering.
- [ ] **Houston unit tests.** Can we keep them as a fixture corpus for the new Go implementation (export them as JSON), or do we re-author?
- [ ] **Performance.** Spec generation today happens in Houston worker pods. After migration, it happens in Commander. Sizing — does the existing commander pod resource request suffice?
- [ ] **Multi-cluster / split-plane.** When CP is on one cluster and DP is on many, Commander runs in each DP cluster. Does each Commander instance get its own copy of defaults? Synced how on chart-version drift?
- [ ] **Backwards compatibility flag lifetime.** How many releases do we keep both paths alive?
- [ ] **Operator-mode adoption** (M2 Task 2): if Commander generates the spec but the CR was adopted, do we still apply, partially apply, or skip? Probably "skip apply by default; targeted apply for fields Houston explicitly modifies" — re-uses the field-ownership work from M2.
